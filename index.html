02 — BACKEND LOGIC
Availability & Slot Locking
The engine that prevents double-bookings and keeps everything in sync.
🟢

"Available Now" Rules

A leader shows as Available Now only when: they're online, have no active session, and their next open slot starts within 5–10 minutes.

🗓️

Slot Generation

Leaders set their daily availability windows. The system auto-generates fixed-duration slots (10/15/30 min) within those windows, with 5-min buffers between sessions.

🔒

Lock Mechanism

On slot selection: status → "locked", locked_by → user_id, lock_expires_at → now + 5 min. Background job checks expiry and releases stale locks.

🔓

Auto-Release

If payment isn't completed within the lock window, a cron/worker flips the slot back to "available." User is notified: "Your slot was released."

Slot Status Lifecycle

available → locked (user selects) → booked (payment done) → completed / missed / cancelled

Approval Edge Cases

Scenario System Behavior
Leader submits overlapping times Client-side validation blocks it. "You already have an entry for this time range."
Ops modifies the time window Leader gets notified: "Your 10 AM–2 PM was adjusted to 11 AM–2 PM. Tap to accept."
Leader doesn't show up during approved window System marks as "no-show." Reliability score drops. Repeated no-shows → auto-deprioritize in queue.
Ops doesn't review within 24 hours Auto-escalation: entry highlighted red in admin panel. Slack/email alert to ops manager.
Leader wants to cancel approved availability If no bookings on those slots → instant cancel. If bookings exist → must contact ops for manual resolution.
Recurring weekly availability First submission needs approval. After 4 consecutive approved weeks with good attendance, auto-approve kicks in for that leader.
04 — SAFEGUARDS
Edge Cases & Fail-Safes
What happens when things don't go as planned. Every scenario needs a defined path.
Scenario What Happens User Sees
User pays, leader unavailable Auto-assign next slot OR suggest alternate leader OR full refund "Leader stepped away. We've moved you to 7:45 PM. Refund if not okay."
User doesn't join Mark as "missed." No refund (or partial based on policy). "You missed your session. Contact support for assistance."
Leader cancels Full refund + apology coupon (10% off next booking) "Your leader cancelled. Full refund issued + 10% off your next session."
Payment fails mid-lock Slot auto-releases after lock expiry. No charge. "Payment didn't go through. Your slot has been released. Try again."
Two users lock same slot First lock wins (DB-level atomic operation). Second user sees "Slot taken." "This slot was just taken. Here are other available times →"
Session runs overtime Soft warning at 2 min left. Hard cut at buffer end. Next session protected. "2 minutes remaining in your session."
05 — DATA MODEL
New Tables to Add
2 new tables for bookings and availability approval tracking. Connected to your existing leaders table via leaderId and your existing tag_slug table via slug.
ℹ️ Your existing tables stay untouched

leaders table — already exists, no changes.
tag_slug table — already exists, no changes.
Both new tables below reference them via foreign keys.

Table 1 — Bookings
bookings -- NEW: tracks every user ↔ leader session booking

  id                UUID       PK
  leaderId          UUID       FK → leaders.id
  userId            UUID       FK → users.id
  slug              VARCHAR    FK → tag_slug.slug -- which service was booked

  ── Slot Details ──
  slotDate          DATE
  slotStartTime     TIME
  slotEndTime       TIME
  durationMin       INT        -- 10, 15, 30

  ── Payment ──
  paymentId         VARCHAR    -- Razorpay/Stripe ref
  paymentStatus     ENUM       -- pending, paid, refunded, failed

  ── Session Tracking ──
  sessionStatus     ENUM       -- upcoming, live, completed, missed, cancelled
  slotLocked        BOOLEAN    -- true while slot is held
  lockExpiresAt     TIMESTAMP -- auto-release trigger
  sessionLink       VARCHAR    -- video call URL (nullable)
  rating            DECIMAL    -- user's post-session rating (nullable)
  review            TEXT       -- user's review text (nullable)

  createdAt         TIMESTAMP
  updatedAt         TIMESTAMP
🔥 Why slug in bookings?

Links to your existing tag_slug table so you know exactly which service type was booked (tarot-reading, astrology, reiki-healing, etc.). Enables filtering bookings by domain: /bookings?slug=tarot-reading and dashboard analytics per service type.

Table 2 — Leader Availability Approvals
leader_availability_approvals -- NEW: ops team approval tracking

  id                UUID       PK
  leaderId          UUID       FK → leaders.id
  slug              VARCHAR    FK → tag_slug.slug -- which service they're offering

  ── Requested Availability ──
  date              DATE
  startTime         TIME
  endTime           TIME
  durationMin       INT        -- session length: 10, 15, 30
  isRecurring       BOOLEAN    -- repeat weekly?

  ── Approval Workflow ──
  status            ENUM       -- pending, approved, rejected, cancelled
  reviewedBy        UUID       -- FK → admin_users (nullable)
  rejectReason      VARCHAR    -- nullable, shown to leader
  modifiedStartTime TIME       -- if ops adjusted the time (nullable)
  modifiedEndTime   TIME       -- if ops adjusted the time (nullable)

  submittedAt       TIMESTAMP -- when leader submitted
  reviewedAt        TIMESTAMP -- when ops acted (nullable)
  createdAt         TIMESTAMP
  updatedAt         TIMESTAMP

🔗 How Everything Connects

leaders (your existing table)
  ├── leaderId ←── bookings
  └── leaderId ←── leader_availability_approvals

tag_slug (your existing table)
  ├── slug ←──── bookings.slug
  └── slug ←──── leader_availability_approvals.slug
📋

Booking Status Flow

pending → slot locked
paid → payment confirmed
live → session in progress
completed → session done
missed → user no-show
cancelled → refund issued

🛡️

Approval Status Flow

pending → leader submitted
approved → ops approved, live on app
rejected → reason shown to leader
